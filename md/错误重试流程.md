# 错误重试流程 v1.9.0+

## 概述

错误处理分三层：
1. **翻译器内部**（openai_hq.py）：处理翻译逻辑错误
2. **并发管道**（concurrent_pipeline.py）：处理批次级错误
3. **并发不强水线管理**（rate_limiter + asyncio事件控制）：管理并发模式、超时、可中断等待

---

## 第一部分：并发槽位机制（v1.9.0+）

### 核心改进

#### 五层架构

1. **修复线程（inpaint worker）：串行独立运行**
   - 一次处理 1 张图片
   - 完全独立，不占用翻译并发槽位

2. **翻译线程（translation worker）：动态并发数控制**
   - 修复运行时：并发 = 1（串行模式，避免 API 过载）
   - 修复完成后：并发 = 用户配置（默认 4）
   - 使用 `rate_limiter.expand_concurrency()` 动态扩展

3. **修复停止事件（inpaint_stopped_event）：可中断的等待机制** ⭐
   - 修复线程完成时：`event.set()` 触发
   - 翻译线程改进：不再是"死等"单个任务，而是 `asyncio.wait(FIRST_COMPLETED)`
   - 二选一逻辑：任务完成 OR 修复停止事件触发，先到先得
   - 结果：修复停止时立即回到调度循环，开始利用新增槽位

4. **速率限制器（rate_limiter）：四层防护**
   - 指数退避：遇到 429 后全局暂停（2/2/4/8s）
   - 并发控制：限制同时发起的请求数（semaphore）
   - 令牌桶：控制每分钟总请求数（RPM）
   - 预热控制：扩展后每 2s 启动一个新请求

5. **流式连接超时（stream timeout）：首字节 40 秒限制** ⭐
   - httpx 配置：`timeout=Timeout(400.0, connect=60.0, read=40.0)`
   - 含义：连接 60s / 读取 40s / 总计 400s
   - 作用：防止服务器断连或长时间无响应的流式请求无限期卡住

#### 实际案例：修复前 vs 修复后

**问题场景**：gemini-3-flash-preview 流式请求卡住 5 分钟（无首字节数据）

**日志时间线（修复前）**：

| 时间 | 事件 | 并发状态 | 问题 |
|------|------|--------|------|
| 23:02:35 | 发送翻译请求（图片 8/8） | 并发=1 | - |
| 23:04:33 | 修复线程完成，event.set() | 扩展到并发=4 | - |
| 23:04:33 ~ 23:07:36 | **翻译线程在 `await task` 上阻塞** | 浪费 3 个槽位！ | 🔴 翻译线程自己卡死 |
| 23:07:36 | 服务器断连（300s 超时） | 仍无法利用新槽位 | 5 分钟白白浪费 |

**根本原因**：翻译线程修复期间采用 `await task` 死等单个批次完成。即使 `inpaint_stopped_event` 触发、并发数扩展到 4，翻译线程仍然被卡在那个"等待中"的任务上，无法继续调度循环去发起新请求。结果：新增的 3 个槽位永远得不到利用。

**修复方案（v1.9.0）**：

```python
# 改进前：死等
if not self.inpaint_stopped_event.is_set():
    await task  # ← 直到 task 完成才返回

# 改进后：可中断等待
if not self.inpaint_stopped_event.is_set():
    wait_inpaint_stop = asyncio.create_task(self.inpaint_stopped_event.wait())
    done, pending = await asyncio.wait(
        {task, wait_inpaint_stop},
        return_when=asyncio.FIRST_COMPLETED,
    )
    # 取消 wait_inpaint_stop（防止悬挂 task）
    if task in done:
        # 任务先完成：清理并移出 running_tasks
        await task
        running_tasks.discard(task)
    # 若 event 先触发：不清理，task 继续占用 1 个槽位运行，翻译线程立即回到调度
```

**修复后效果**：
- 修复停止后，翻译线程立即被唤醒
- 不再阻塞在单个批次上
- 立刻开始利用新增的 3 个槽位去发起后续请求
- 卡住的请求继续占用 1 个槽位跑（直到超时/完成）

## 第二部分：流式连接超时限制（v1.9.0+）

### 首字节超时配置

**位置**：`manga_translator/translators/openai_hq.py` 第 192 行

```python
http_client=httpx.AsyncClient(
    headers=BROWSER_HEADERS,
    timeout=httpx.Timeout(400.0, connect=60.0, read=40.0)
    #                ↑              ↑              ↑
    #            总超时=400s   连接=60s      首字节/块=40s
)
```

### 工作原理

- **connect=60s**：建立 TCP 连接的最大时间
- **read=40s**：在流式模式下，每次读取 chunk 的最大时间
  - 若 40s 内没收到 chunk → 超时中断
  - 包括首字节超时 + 后续块超时
- **timeout=400s**：整个请求的总超时时间

### 修复前 vs 修复后

**修复前**：
- 配置：`timeout=Timeout(300.0, connect=60.0)`（无 read 限制）
- 行为：流式请求 300s 内没收到数据也不中断（只看总时间）
- 问题：服务器缓慢响应时卡住 5+ 分钟

**修复后**：
- 配置：`timeout=Timeout(400.0, connect=60.0, read=40.0)`
- 行为：40s 内没收到数据就超时中断
- 好处：最多等 40s 就放弃，而不是等 5 分钟

## 第三部分：429 限速错误独立处理（v1.9.0+）

### 独立计数与指数退避

429 限速错误不受 `max_retries` 限制，单独计数：

| 尝试次数 | 退避时间 | 总耗时 |
|---------|--------|-------|
| 1（初始） | - | - |
| 2 | 2 秒 | 2s |
| 3 | 4 秒 | 6s |
| 4 | 8 秒 | 14s |
| 5（失败） | - | 放弃 |

### 全局限速通知

```python
# 一旦触发 429，通知全局限速器
ctx.rate_limiter.report_rate_limit_error()
  → 所有其他并发请求进入退避状态
  → 新请求不发出，直到退避结束
```

## 第四部分：翻译器内部重试（openai_hq.py）

这些错误在单次 API 调用内部处理，**不涉及拆分**。

### 重试计数机制

`attempt` 仅在 while 循环顶部递增一次（每次循环 = 一次 API 调用）。
失败路径（空内容/except/400等）不再额外 `attempt += 1`，避免双重计数。

`max_retries=2` 时实际执行 2 次 API 调用，日志显示 `(1/2)` → `(2/2)` → 错误。

### raise 被自身 except 捕获问题

try 块内的 `raise last_exception`（max_retries 耗尽时）会被同函数的 `except Exception` 捕获。
通过 `_retries_exhausted` 标志位解决：raise 前设 `_retries_exhausted = True`，except 入口检测到直接 re-raise，不再重复计数/打印日志。

### 重试时客户端重建（并发安全）

每次重试前调用 `_rebuild_client_for_retry()` 重建 API 客户端连接。

**为什么不用 `_setup_client(force_recreate=True)`**：
- 并发管道中多个 worker 共享同一个翻译器实例的 `self.client`
- `force_recreate` 会 `close()` 旧客户端，断开其他 worker 正在使用的 in-flight 连接

**`_rebuild_client_for_retry()` 的做法**：
- 只创建新客户端赋值给 `self.client`，不关闭旧的
- 旧客户端由 in-flight 协程持有引用，请求完成后 GC 自动释放
- 当前 worker 下次循环使用新客户端，其他 worker 的 in-flight 请求不受影响

```
Worker A: self.client → client_1 → 创建协程 → await (in-flight)
Worker B: 失败 → _rebuild_client_for_retry() → self.client = client_2
          client_1 不关闭，Worker A 的协程仍持有引用
Worker A: 请求完成 → client_1 无引用 → GC 回收
Worker A: 下次循环 → self.client = client_2 → 用新连接
```

**哪些重试路径会重建客户端**：
- ✅ 数量不匹配、质量检查失败、BR标记缺失（质量重试）
- ✅ 空内容/content_filter/length/tool_calls（max_retries 重试）
- ✅ 400错误、其他通用错误（except 块重试）
- ❌ 429限速错误不重建（连接正常，只是服务端限流）

### 1. 数量不匹配
**触发条件**：翻译结果数量 ≠ 原文数量

**重试策略**（固定 2 次，与 max_retries 独立）：
```
API返回 → 解析结果 → 数量不匹配
       → 重建客户端 → 等待2秒 → 原样重试（同一批次）
       → 最多重试 2 次（含初始=共2次机会）
       → 失败则抛出异常，由并发管道加入延迟队列
```

### 2. 质量检查失败
**触发条件**：空翻译、合并翻译、可疑符号等

**重试策略**（固定 2 次，与 max_retries 独立）：
```
API返回 → 质量验证失败
       → 重建客户端 → 等待2秒 → 提高温度重试
       → 最多重试 2 次（含初始=共2次机会）
       → 失败则抛出异常，由并发管道加入延迟队列
```

### 3. BR标记缺失
**触发条件**：多行文本翻译缺少 [BR] 换行标记

**重试策略**（固定 2 次，与 max_retries 独立）：
```
API返回 → BR检查失败
       → 重建客户端 → 等待2秒 → 提高温度重试
       → 最多重试 2 次（含初始=共2次机会）
       → 失败则抛出异常，由并发管道加入延迟队列
```

### 4. 429 限速错误
**触发条件**：API 返回 429 Too Many Requests

**重试策略**（独立计数，不受 max_retries 限制，不重建客户端）：
```
触发429 → 通知全局限速器(report_rate_limit_error)
       → 指数退避 2^n 秒（最大 64 秒）
       → 重试（不重建客户端，连接正常）
       → 总共尝试 4 次（初始 1 次 + 重试 3 次）
```

**退避时间**：
| 尝试次数 | 退避时间 |
|---------|--------|
| 1 | 2 秒 |
| 2 | 2 秒 |
| 3 | 4 秒 |
| 4 | 8 秒 |

### 5. 空内容/content_filter/length/tool_calls
**触发条件**：API 返回空内容、`finish_reason` 为 content_filter/length/tool_calls 等

**重试策略**（受 max_retries 控制）：
```
API返回空内容/被拦截
  → 记录警告，标记下次不发送图片（降级）
  → 重建客户端 → 等待1秒 → 重试
  → 达到 max_retries 后抛出异常（由 _translate_with_split 处理降级/拆分）
```

### 6. 其他错误（except 块）
**触发条件**：400错误、网络错误、其他未分类异常

**重试策略**（受 max_retries 控制）：
```
异常被 except 捕获
  → 多模态不支持(400) → 直接抛出，不重试
  → 其他400错误 → 重建客户端 → 等待1秒 → 重试
  → 429限速 → 独立计数（见上方）
  → 502等网络错误 → 标记不发送图片 + 重建客户端 → 等待1秒 → 重试
  → 达到 max_retries 后抛出异常
```

## 第五部分：并发管道重试（concurrent_pipeline.py）

这些错误在批次级别处理，可能涉及**拆分重试**。

### 1. 超时错误（延迟队列 + 拆分重试）
**触发条件**：翻译器抛出超时异常（例如 `TimeoutError` / `asyncio.TimeoutError` / `httpx.TimeoutException` / `RemoteProtocolError`）

**说明**：
- OpenAI/OpenAI HQ 翻译器侧超时配置：**400 秒**，且**仅允许超时 1 次**（翻译器内部不做超时重试）
- 首字节超时：**40 秒**（流式连接独有）
- 超时不会触发"不发送图片"的降级逻辑（超时只做失败上抛）
- 并发管道侧策略：**不做整批重试**，直接进入延迟队列，避免阻塞主队列槽位

**处理策略**：
```
批次[A,B,C] 超时
  → 将 A/B/C 逐张加入 retry_queue（error_type=timeout_error）
  → 立即返回，主队列继续处理下一批
  → 主队列完成后，_process_retry_queue() 并行重试（单张图片重试 1 次，受 concurrency/RPM 控制）
```

### 2. 网络错误（整批重试 → 延迟队列拆分重试）
**触发条件**：
- 连接断开、重置
- 524 服务器超时
- 504 网关超时 / Cloudflare 5xx（常见表现：返回 HTML 错误页）
- socket/httpcore/httpx 等网络异常
- **流式连接中断**（RemoteProtocolError、peer closed connection 等）
- **响应内容为 HTML 错误页**（即使 HTTP 状态码不明显/未抛出网络异常）

**关键词匹配**（不含 timeout；timeout 单独归类为"超时错误"）：
```
connect, reset, closed, eof, broken pipe,
connection, network, socket, httpcore, httpx,
504, 520, 522, 523, 524,
gateway timeout, gateway time-out,
cloudflare, cf-error, cf-ray, <!DOCTYPE html>,
remoteprotocolerror, peer closed connection,
incomplete message body, server disconnected
```

**说明**：
- 这类问题有时不是"抛异常"，而是"返回了一段错误 HTML"。因此需要在**响应验证阶段**识别（例如 `<!DOCTYPE html>` / `cloudflare`），并统一按 `network_error` 走"整批重试 → 失败后入延迟队列"策略。

**处理策略**：
```
批次[A,B,C] 网络错误
  → 等待3秒 → 整批重试 1 次（skip_rate_limiter=True，不额外等待 RPM 令牌）
  → 若仍失败 → 将 A/B/C 逐张加入 retry_queue（error_type=network_error）
  → 主队列完成后，_process_retry_queue() 并行重试（单张图片重试 1 次，受 concurrency/RPM 控制）
```

### 3. 401认证错误（整批重试 → 延迟队列拆分重试）
**触发条件**：API 返回 401 Unauthorized

**关键词匹配**：
```
401, unauthorized, authentication
```

**处理策略**：
```
批次[A,B,C] 401
  → 等待2秒 → 整批重试 1 次（skip_rate_limiter=True，不额外等待 RPM 令牌）
  → 若仍失败 → 将 A/B/C 逐张加入 retry_queue（error_type=auth_error）
  → 主队列完成后，_process_retry_queue() 并行重试（单张图片重试 1 次，受 concurrency/RPM 控制）
```

### 4. 安全限制/数量不匹配/BR缺失/质量检查失败（延迟队列拆分重试）
**触发条件**：
- 安全限制/内容过滤
- 数量不匹配（翻译器抛出 count mismatch）
- BR标记缺失（翻译器抛出 BRMarkersValidationException）
- 质量检查失败（翻译器抛出 quality failed）

**关键词匹配**：
```
safety, content, filter, block, policy, violation,
inappropriate, harmful, moderation, flagged, content_filter
```

**注意**：翻译器内部的 `content_filter`（finish_reason）也会抛出到此层处理

**重试策略**（concurrent_pipeline 层）：
```
批次[A,B,C] 失败 → 加入延迟重试队列
  → 立即释放槽位，不阻塞主队列
  → 主队列完成后，统一并行处理延迟队列
    A → 使用速率限制器 → 重试1次 → 成功/降级
    B → 使用速率限制器 → 重试1次 → 成功/降级
    C → 使用速率限制器 → 重试1次 → 成功/降级
```

**延迟重试队列优势**：
- ✅ 不阻塞主翻译队列的并发槽位
- ✅ 主队列完成后再处理，整体吞吐量更高
- ✅ 並行处理，受 `concurrency` 配置控制

**降级策略**（manga_translator 层，HQ翻译器单张失败时）：
```
单张HQ翻译失败 → 检测是否安全限制错误
  → 若是安全限制 → 降级到文本翻译（不发送图片，批量处理）
    openai_hq → openai
    gemini_hq → gemini
  → 若非安全限制 → 直接标记失败
```
- 延迟到主队列完成后处理
- 使用并发控制，不会过载 API
- 部分成功、部分失败均可
- 降级翻译使用批量处理（不是逐条）

### 5. 其他错误（直接失败）
**触发条件**：不属于以上类别的错误

**处理策略**：
```
批次[A,B,C] 失败 → 全部标记失败 → 跳过渲染
```
- 不重试
- 记录到 `failed_images` 列表

### 6. 用户取消（加入失败队列）
**触发条件**：用户点击"停止"按钮取消任务

**处理策略**：
```
用户取消 → asyncio.CancelledError
  → process_batch 捕获取消信号
  → 设置 stop_workers = True，停止所有工作线程
  → 收集已处理的图片（渲染成功 + 已失败）
  → 将未处理的图片加入 failed_images（错误信息="用户取消"）
  → 返回合并结果（成功 + 失败 + 取消）
```

**失败队列记录**：
- ✅ 取消的图片加入 `failed_images` 列表
- ✅ 转换为 `Context` 对象并返回（`success=False`, `translation_error="用户取消"`）
- ✅ UI 层统计时正确区分成功/失败/取消

**日志示例**：
```
[并发流水线] 任务被取消，已处理 127 张，未处理 69 张
[并发流水线] ⚠️ 以下 69 张图片未完成:
  ❌ 022.jpg: 用户取消
  ❌ 023.png: 用户取消
```

## 第六部分：HQ 批量翻译空内容降级机制（v1.9.x+）

### 触发条件
当 openai_hq 批量翻译返回空内容时（HTTP 200 但 `finish_reason: stop` 且无实际内容），触发该降级流程。

**错误关键词匹配**：
```
returned empty content, api returned empty text,
返回空内容, 返回空文本
```

### 三级降级策略

```
第一级：HQ 批量翻译（N 张图片）
    │
    └── 空内容错误 → 进入第二级
                       │
第二级：HQ 单图翻译（逐张重试）
    │
    ├── 成功 → 继续下一张
    └── 失败 → 进入第三级
                       │
第三级：openai 文本翻译（不发送图片）
    │
    ├── 成功 → 继续下一张
    └── 失败 → 返回原文（不中断流程）
```

### 处理流程详解

**代码位置**: `manga_translator/manga_translator.py` → `_batch_translate_texts()`

```
1. 检测到 openai_hq 返回空内容
   ↓
2. 从 ctx.high_quality_batch_data 获取批次数据
   ↓
3. 按图片拆分为单图任务
   ↓
4. 对每张图片：
   ├── 构建 single_ctx（复制 image_name、from_lang、术语等）
   ├── 调用 HQ 单图翻译
   └── 若失败，调用 _fallback_batch_text_translate()
   ↓
5. 合并结果返回
```

### 上下文传递

单图重试时，会从原始批次上下文复制以下关键字段：

| 字段 | 用途 |
|------|------|
| `image_name` | 作品名识别，用于术语匹配 |
| `from_lang` | 源语言，用于加载语言专用提示词 |
| `custom_prompt_json` | 已加载的术语和自定义提示 |
| `glossary_write_work_name` | 术语写入目标作品 |
| `raw_work_name` | 原始生肉作品名 |
| `work_name` | 映射后的熟肉作品名 |
| `local_prev_context` | 同批次前序图片原文上下文 |

### 与安全限制降级的区别

| 特性 | 空内容降级 | 安全限制降级 |
|------|----------|------------|

## 第七部分：延迟重试队列机制（v1.9.x+）

### 设计目的
某些错误（超时、安全限制、数量不匹配、BR缺失、质量失败等）如果在主队列里反复重试，会阻塞并发槽位，影响整体吞吐量。
延迟重试队列将这类错误推迟到主队列完成后处理，并在延迟队列阶段按“单张图片”拆分重试。

### 新增机制（Worker并行消费）
- 延迟队列从 `list` 改为 `asyncio.Queue`，支持多 Worker 并行消费。
- 主队列结束后，不再由翻译线程统一调用 `_process_retry_queue()`；而是 **Worker 收到“切换信号”后自行处理延迟队列**。
- **切换信号只在延迟队列不为空时发送**；如果延迟队列为空，则发送退出信号让 Worker 直接结束。

### 工作流程（更新版）
```
主翻译队列处理中...
  ↓
批次遇到可延迟错误（超时/安全限制/数量不匹配/BR缺失/质量失败/网络错误整批重试失败/401整批重试失败）
  ↓
加入 retry_queue（不立即重试）
  ↓
释放槽位，继续处理下一批次
  ↓
收集结束，检查延迟队列是否为空
  ├─ 为空：发送退出信号 → Worker 退出
  └─ 非空：发送切换信号 → Worker 进入延迟队列并行重试
  ↓
延迟队列清空 → Worker 退出
```

### 延迟队列启动条件（v1.9.x 优化）

**问题背景**：在早期版本中，延迟队列会在主翻译队列"暂时为空"时立即启动。但如果 OCR/检测还在持续产生新任务，延迟队列的重试会阻塞正常的串行翻译恢复，导致吞吐量下降。

**优化方案**：延迟队列只在主流程**完全结束**后才启动处理。

**触发条件**（两者必须同时满足）：
1. `detection_ocr_done = True`（所有图片的检测+OCR 已完成）
2. `translation_queue.empty()`（主翻译队列为空）

**守卫逻辑**（Worker 侧）：
```python
if batch is _RETRY_SWITCH:
    # 守卫：确保主流程完全结束后再处理延迟队列
    if (not self.detection_ocr_done) or (not self.translation_queue.empty()):
        batch_queue.task_done()
        await batch_queue.put(_RETRY_SWITCH)  # 放回切换信号
        await asyncio.sleep(0.2)              # 短暂等待
        continue                               # 继续主循环
    # ... 进入延迟队列处理
```

**效果**：
- ✅ 主流程（检测+OCR → 翻译）不被延迟队列阻塞
- ✅ 串行翻译模式可正常恢复
- ✅ 延迟队列在所有图片处理完后统一重试

### 延迟队列处理逻辑（Worker模式）
```
Worker 收到切换信号后：
  while 延迟队列未空：
    取出一条 (ctx, config, error_type)
    执行 _retry_single_image()
  队列空且所有主队列Worker结束 → 退出
```

### 上下文记录方式
- 入延迟队列时保存的是 `(ctx, config, error_type)`。
- `ctx` 为**当前上下文对象引用**，不是深拷贝快照。
- 如果需要冻结当时状态，需要额外 `deepcopy`（内存开销较大）。

### 哪些错误使用延迟队列

| 错误类型 | 使用延迟队列 | 说明 |
|---------|----------|------|
| 超时错误（timeout_error） | ✅ 是 | 直接进入延迟队列，按单张图片拆分重试 |
| 安全限制/内容过滤（safety_limit） | ✅ 是 | 直接进入延迟队列，按单张图片拆分重试 |
| 数量不匹配（count_mismatch） | ✅ 是 | 翻译器内部重试后直接进入延迟队列（保持整批，延迟队列中先整批重试，再单张重试） |
| BR标记缺失（br_missing） | ✅ 是 | 翻译器内部重试后直接进入延迟队列（保持整批，延迟队列中先整批重试，再单张重试） |
| 质量检查失败（quality_failed） | ✅ 是 | 翻译器内部重试后直接进入延迟队列（保持整批，延迟队列中先整批重试，再单张重试） |
| HQ空内容（empty_content） | ✅ 是 | 直接进入延迟队列，按单张图片拆分重试（可触发HQ→文本降级） |
| 网络错误（network_error） | ✅ 视情况 | 整批重试 1 次仍失败后进入延迟队列（按单张图片）；包含 504/Cloudflare/HTML 错误页 |
| 401认证错误（auth_error） | ✅ 视情况 | 整批重试 1 次仍失败后进入延迟队列（按单张图片） |

## 第七部分：速率限制机制

### 四层防护
1. **指数退避** - 遇到 429 后全局暂停
2. **预热控制** - 并发扩展后渐进式启动（每 2 秒启动一个）
3. **并发控制** - 限制同时发出的请求数
4. **令牌桶** - 控制每分钟总请求数（RPM）

### 重试的特殊处理
主队列内的错误重试**不等待令牌**：
- ❌ 不等待 RPM 令牌
- ❌ 不占用新并发槽位
- ✅ 仍占用原槽位（在同一调用栈内）

延迟队列的重试**使用速率限制器**：
- ✅ 等待 RPM 令牌
- ✅ 受并发数限制
- ✅ 主队列完成后执行，不影响吞吐量

## 重试次数总结

| 错误类型 | 主队列处理 | 延迟队列处理 | 退避时间 |
|---------|----------|-----------|---------|
| 超时（timeout_error） | 不整批重试，直接入队 | 单张图片重试 1 次 | -（单次请求超时=400s） |
| 网络错误（network_error） | 整批重试 1 次（含初始=2次） | 单张图片重试 1 次（仅在整批仍失败时） | 3 秒（包含 504/Cloudflare/HTML 错误页） |
| 401认证（auth_error） | 整批重试 1 次（含初始=2次） | 单张图片重试 1 次（仅在整批仍失败时） | 2 秒 |
| 安全限制/内容过滤（safety_limit） | 直接入队 | 单张图片重试 1 次（可能触发 HQ→文本降级） | - |
| 数量不匹配（count_mismatch） | 翻译器内部重试后直接入队 | 整批重试 1 次 → 单张重试 1 次（可触发 HQ→文本降级） | - |
| 质量检查失败（quality_failed） | 翻译器内部重试后直接入队 | 整批重试 1 次 → 单张重试 1 次（可触发 HQ→文本降级） | - |
| BR标记缺失（br_missing） | 翻译器内部重试后直接入队 | 整批重试 1 次 → 单张重试 1 次（可触发 HQ→文本降级） | - |
| **429限速** | 翻译器内部独立重试（总共4次） | - | 2/2/4/8 秒 |
| **HQ空内容（empty_content）** | 直接入队 | 单张图片重试 1 次（可触发 HQ→文本降级） | - |
| 其他错误 | 直接失败或由 max_retries 决定 | - | 1 秒（若有） |
| **用户取消** | 不重试，加入 failed_images | - | - |
