# 错误重试流程

## 概述

错误处理分两层：
1. **翻译器内部**（openai_hq.py）：处理翻译逻辑错误
2. **并发管道**（concurrent_pipeline.py）：处理批次级错误

---

## 重试次数配置（max_retries）

`max_retries` = 配置文件中的 `translator.attempts` 参数

| 值 | 含义 |
|---|---|
| `-1` | 无限重试 |
| `N` | 总共尝试 N 次（初始 1 次 + 重试 N-1 次） |

**当前值**：`2`（初始 1 次 + 重试 1 次）

**配置位置**：
- 配置文件：`config.json` → `translator.attempts`
- 命令行：`--attempts N`

---

## 一、翻译器内部重试（openai_hq.py）

这些错误在单次 API 调用内部处理，**不涉及拆分**。

### 1. 数量不匹配
**触发条件**：翻译结果数量 ≠ 原文数量

**重试策略**：
```
API返回 → 解析结果 → 数量不匹配
       → 等待2秒 → 原样重试（同一批次）
       → 最多重试至 max_retries 次
       → 失败则抛出异常
```

### 2. 质量检查失败
**触发条件**：空翻译、合并翻译、可疑符号等

**重试策略**：
```
API返回 → 质量验证失败
       → 等待2秒 → 提高温度重试
       → 最多重试至 max_retries 次
       → 失败则抛出异常
```

### 3. BR标记缺失
**触发条件**：多行文本翻译缺少 [BR] 换行标记

**重试策略**：
```
API返回 → BR检查失败
       → 等待2秒 → 提高温度重试
       → 最多重试至 max_retries 次
       → 失败则抛出异常
```

### 4. 429 限速错误
**触发条件**：API 返回 429 Too Many Requests

**重试策略**（独立计数，不受 max_retries 限制）：
```
触发429 → 通知全局限速器(report_rate_limit_error)
       → 指数退避 2^n 秒（最大 64 秒）
       → 重试
       → 总共尝试 4 次（初始 1 次 + 重试 3 次）
```

**退避时间**：
| 尝试次数 | 退避时间 |
|---------|--------|
| 1 | 2 秒 |
| 2 | 2 秒 |
| 3 | 4 秒 |
| 4 | 8 秒 |

### 5. 内容过滤（content_filter）
**触发条件**：API 返回 `finish_reason = 'content_filter'`

**处理策略**：
```
触发过滤 → 直接抛出异常
        → 由 concurrent_pipeline 捕获并归类为 safety_limit
        → 加入 retry_queue（按单张图片）
        → 主队列完成后在 _process_retry_queue() 阶段重试
```

## 二、并发管道重试（concurrent_pipeline.py）

这些错误在批次级别处理，可能涉及**拆分重试**。

### 1. 超时错误（延迟队列 + 拆分重试）
**触发条件**：翻译器抛出超时异常（例如 `TimeoutError` / `asyncio.TimeoutError` / `httpx.TimeoutException`）

**说明**：
- OpenAI/OpenAI HQ 翻译器侧超时配置：**400 秒**，且**仅允许超时 1 次**（翻译器内部不做超时重试）
- 超时不会触发“不发送图片”的降级逻辑（超时只做失败上抛）
- 并发管道侧策略：**不做整批重试**，直接进入延迟队列，避免阻塞主队列槽位

**处理策略**：
```
批次[A,B,C] 超时
  → 将 A/B/C 逐张加入 retry_queue（error_type=timeout_error）
  → 立即返回，主队列继续处理下一批
  → 主队列完成后，_process_retry_queue() 并行重试（单张图片重试 1 次，受 concurrency/RPM 控制）
```

### 2. 网络错误（整批重试 → 延迟队列拆分重试）
**触发条件**：
- 连接断开、重置
- 524 服务器超时
- 504 网关超时 / Cloudflare 5xx（常见表现：返回 HTML 错误页）
- socket/httpcore/httpx 等网络异常
- **响应内容为 HTML 错误页**（即使 HTTP 状态码不明显/未抛出网络异常）

**关键词匹配**（不含 timeout；timeout 单独归类为“超时错误”）：
```
connect, reset, closed, eof, broken pipe,
connection, network, socket, httpcore, httpx,
504, 520, 522, 523, 524,
gateway timeout, gateway time-out,
cloudflare, cf-error, cf-ray, <!DOCTYPE html>
```

**说明**：
- 这类问题有时不是“抛异常”，而是“返回了一段错误 HTML”。因此需要在**响应验证阶段**识别（例如 `<!DOCTYPE html>` / `cloudflare`），并统一按 `network_error` 走“整批重试 → 失败后入延迟队列”策略。

**处理策略**：
```
批次[A,B,C] 网络错误
  → 等待3秒 → 整批重试 1 次（skip_rate_limiter=True，不额外等待 RPM 令牌）
  → 若仍失败 → 将 A/B/C 逐张加入 retry_queue（error_type=network_error）
  → 主队列完成后，_process_retry_queue() 并行重试（单张图片重试 1 次，受 concurrency/RPM 控制）
```

### 3. 401认证错误（整批重试 → 延迟队列拆分重试）
**触发条件**：API 返回 401 Unauthorized

**关键词匹配**：
```
401, unauthorized, authentication
```

**处理策略**：
```
批次[A,B,C] 401
  → 等待2秒 → 整批重试 1 次（skip_rate_limiter=True，不额外等待 RPM 令牌）
  → 若仍失败 → 将 A/B/C 逐张加入 retry_queue（error_type=auth_error）
  → 主队列完成后，_process_retry_queue() 并行重试（单张图片重试 1 次，受 concurrency/RPM 控制）
```

### 4. 安全限制/数量不匹配/BR缺失/质量检查失败（延迟队列拆分重试）
**触发条件**：
- 安全限制/内容过滤
- 数量不匹配（翻译器抛出 count mismatch）
- BR标记缺失（翻译器抛出 BRMarkersValidationException）
- 质量检查失败（翻译器抛出 quality failed）

**关键词匹配**：
```
safety, content, filter, block, policy, violation,
inappropriate, harmful, moderation, flagged, content_filter
```

**注意**：翻译器内部的 `content_filter`（finish_reason）也会抛出到此层处理

**重试策略**（concurrent_pipeline 层）：
```
批次[A,B,C] 失败 → 加入延迟重试队列
  → 立即释放槽位，不阻塞主队列
  → 主队列完成后，统一并行处理延迟队列
    A → 使用速率限制器 → 重试1次 → 成功/降级
    B → 使用速率限制器 → 重试1次 → 成功/降级
    C → 使用速率限制器 → 重试1次 → 成功/降级
```

**延迟重试队列优势**：
- ✅ 不阻塞主翻译队列的并发槽位
- ✅ 主队列完成后再处理，整体吞吐量更高
- ✅ 并行处理，受 `concurrency` 配置控制

**降级策略**（manga_translator 层，HQ翻译器单张失败时）：
```
单张HQ翻译失败 → 检测是否安全限制错误
  → 若是安全限制 → 降级到文本翻译（不发送图片，批量处理）
    openai_hq → openai
    gemini_hq → gemini
  → 若非安全限制 → 直接标记失败
```
- 延迟到主队列完成后处理
- 使用并发控制，不会过载 API
- 部分成功、部分失败均可
- 降级翻译使用批量处理（不是逐条）

### 5. 其他错误（直接失败）
**触发条件**：不属于以上类别的错误

**处理策略**：
```
批次[A,B,C] 失败 → 全部标记失败 → 跳过渲染
```
- 不重试
- 记录到 `failed_images` 列表

### 6. 用户取消（加入失败队列）
**触发条件**：用户点击“停止”按钮取消任务

**处理策略**：
```
用户取消 → asyncio.CancelledError
  → process_batch 捕获取消信号
  → 设置 stop_workers = True，停止所有工作线程
  → 收集已处理的图片（渲染成功 + 已失败）
  → 将未处理的图片加入 failed_images（错误信息="用户取消"）
  → 返回合并结果（成功 + 失败 + 取消）
```

**失败队列记录**：
- ✅ 取消的图片加入 `failed_images` 列表
- ✅ 转换为 `Context` 对象并返回（`success=False`, `translation_error="用户取消"`）
- ✅ UI 层统计时正确区分成功/失败/取消

**日志示例**：
```
[并发流水线] 任务被取消，已处理 127 张，未处理 69 张
[并发流水线] ⚠️ 以下 69 张图片未完成:
  ❌ 022.jpg: 用户取消
  ❌ 023.png: 用户取消
```

## 三、延迟重试队列机制（v1.7.5+）

### 设计目的
某些错误（超时、安全限制、数量不匹配、BR缺失、质量失败等）如果在主队列里反复重试，会阻塞并发槽位，影响整体吞吐量。
延迟重试队列将这类错误推迟到主队列完成后统一处理，并在延迟队列阶段按“单张图片”拆分重试。

### 工作流程
```
主翻译队列处理中...
  ↓
批次遇到可延迟错误（超时/安全限制/数量不匹配/BR缺失/质量失败/网络错误整批重试失败/401整批重试失败）
  ↓
加入 retry_queue（不立即重试）
  ↓
释放槽位，继续处理下一批次
  ↓
主队列全部完成
  ↓
_process_retry_queue() 并行处理延迟队列
  ↓
所有图片处理完成
```

### 延迟队列处理逻辑
```python
# 并行处理，受 concurrency 配置控制
running_tasks = set()
for (ctx, config, error_type) in retry_queue:
    while len(running_tasks) >= concurrency:
        # 等待某个任务完成
        done, running_tasks = await asyncio.wait(running_tasks, return_when=FIRST_COMPLETED)
    
    # 启动新的重试任务
    task = asyncio.create_task(_retry_single_image(ctx, config, error_type))
    running_tasks.add(task)

# 等待所有任务完成
await asyncio.gather(*running_tasks)
```

### 哪些错误使用延迟队列
|| 错误类型 | 使用延迟队列 | 说明 |
||---------|------------|------|
|| 超时错误（timeout_error） | ✅ 是 | 直接进入延迟队列，按单张图片拆分重试 |
|| 安全限制/内容过滤（safety_limit） | ✅ 是 | 直接进入延迟队列，按单张图片拆分重试 |
|| 数量不匹配（count_mismatch） | ✅ 是 | 翻译器内部重试失败后进入延迟队列 |
|| BR标记缺失（br_missing） | ✅ 是 | 翻译器内部重试失败后进入延迟队列 |
|| 质量检查失败（quality_failed） | ✅ 是 | 翻译器内部重试失败后进入延迟队列 |
|| 网络错误（network_error） | ✅ 视情况 | 整批重试 1 次仍失败后进入延迟队列（按单张图片）；包含 504/Cloudflare/HTML 错误页 |
|| 401认证错误（auth_error） | ✅ 视情况 | 整批重试 1 次仍失败后进入延迟队列（按单张图片） |

## 四、速率限制机制

### 四层防护
1. **指数退避** - 遇到 429 后全局暂停
2. **预热控制** - 并发扩展后渐进式启动（每 2 秒启动一个）
3. **并发控制** - 限制同时发出的请求数
4. **令牌桶** - 控制每分钟总请求数（RPM）

### 重试的特殊处理
主队列内的错误重试**不等待令牌**：
- ❌ 不等待 RPM 令牌
- ❌ 不占用新并发槽位
- ✅ 仍占用原槽位（在同一调用栈内）

延迟队列的重试**使用速率限制器**：
- ✅ 等待 RPM 令牌
- ✅ 受并发数限制
- ✅ 主队列完成后执行，不影响吞吐量

## 五、完整流程

### 阶段 1：准备与资源获取

**步骤 1.1 - 翻译批次初始化**
- 接收批次数据（包含 1-N 张图片）
- 初始化计数器：`attempt = 0`, `_rate_limit_attempt = 0`
- 设置 `max_retries` = 2（当前配置）

**步骤 1.2 - 速率限制检查**
- 检查全局 429 退避状态
- 如果处于退避期 → 等待
- 如果在预热期 → 串行启动（间隔 2 秒）

**步骤 1.3 - 获取并发槽位**
- 调用 `rate_limiter.acquire()`
- 等待可用的并发槽位（semaphore）
- 消耗 1 个 RPM 令牌（token bucket）
- **资源占用成功**，进入下一步

---

### 阶段 2：API 调用

**步骤 2.1 - 构建请求**
- 准备图片（编码为 base64）
- 在图片上绘制带编号的文本框
- 构建系统提示词和用户提示词
- 如果是重试 → 添加重试提示，调整温度

**步骤 2.2 - 发送 API 请求**
```
attempt += 1  # 计数器 +1
API 请求 → 等待响应
```

**步骤 2.3 - 等待响应**
- 正常响应 → 进入阶段 3
- 异常 → 进入阶段 4（错误处理）

---

### 阶段 3：响应验证（成功路径）

**步骤 3.1 - 基础验证**
- ✅ 检查响应对象有效
- ✅ 检查 `finish_reason` 不是 `content_filter`
- ✅ 检查返回内容不为空
- ✅ 检查不是 HTML 错误页面（例如 404 / Cloudflare 504）
  - 若返回内容疑似错误页（例如包含 `<!DOCTYPE html>` / `cloudflare` / `cf-error`），应按 **网络错误（network_error）** 处理（见“分支 4.7 - 网络错误”）

**步骤 3.2 - 解析翻译结果**
- 去除 `<think>` 标签（推理过程）
- 提取 `<answer>` 内容（如果有）
- 解析翻译和术语

**步骤 3.3 - 数量验证**
- 检查：翻译数量 = 原文数量？
- ❌ 不匹配 → 进入**步骤 4.1（数量不匹配重试）**
- ✅ 匹配 → 继续

**步骤 3.4 - 质量验证**
- 检查空翻译、合并翻译、可疑符号
- ❌ 质量不合格 → 进入**步骤 4.2（质量检查重试）**
- ✅ 质量合格 → 继续

**步骤 3.5 - BR 标记验证**
- 检查多行文本是否包含必要的 `[BR]` 换行标记
- ❌ BR 缺失 → 进入**步骤 4.3（BR 检查重试）**
- ✅ BR 正确 → 继续

**步骤 3.6 - 翻译成功**
```
_rate_limit_attempt = 0  # 重置 429 计数器
return translations  # 返回翻译结果
→ 进入阶段 5（资源释放）
```

### 阶段 4：错误处理与重试

#### 分支 4.1 - 数量不匹配
```
retry_attempt += 1
retry_reason = "Translation count mismatch"

日志：[attempt/max_retries] 数量不匹配，期望 X 条，得到 Y 条

IF attempt >= max_retries (2):
    抛出异常 → 失败
ELSE:
    等待 2 秒 → 返回步骤 2.1（原样重试）
```

#### 分支 4.2 - 质量检查失败
```
retry_attempt += 1
retry_reason = "Quality check failed: {具体原因}"
温度 += 0.2  # 提高温度，帮助跳出错误模式

日志：[attempt/max_retries] 质量检查失败，提高温度重试

IF attempt >= max_retries (2):
    抛出异常 → 失败
ELSE:
    等待 2 秒 → 返回步骤 2.1（提高温度重试）
```

#### 分支 4.3 - BR 标记缺失
```
retry_attempt += 1
retry_reason = "BR markers missing"
温度 += 0.2

日志：[attempt/max_retries] BR 检查失败，提高温度重试

IF attempt >= max_retries (2):
    抛出 BRMarkersValidationException → 失败
ELSE:
    等待 2 秒 → 返回步骤 2.1（提高温度重试）
```

#### 分支 4.4 - 429 限速错误（独立计数）
```
_rate_limit_attempt += 1  # 独立计数器 +1
max_rate_limit_attempts = 4

wait_time = 2^_rate_limit_attempt  # 指数退避
IF API 返回 "retry after N":
    wait_time = N  # 使用 API 建议时间

日志：[限速] 触发 429 限速错误 (X/4)
日志：[限速] 等待 {wait_time}s 后重试...

rate_limiter.report_rate_limit_error()  # 通知全局限速器
→ 全局进入退避状态，影响所有并发请求

IF _rate_limit_attempt >= 4:
    _rate_limit_attempt = 0  # 重置计数器
    抛出 RateLimitError → 失败
ELSE:
    等待 {wait_time} 秒 → 返回步骤 2.1（不计入 max_retries）

退避时间表：
  尝试1 → 等待 2 秒
  尝试2 → 等待 2 秒
  尝试3 → 等待 4 秒
  尝试4 → 等待 8 秒
```

#### 分支 4.5 - 内容过滤（content_filter）
```
直接抛出异常: "content_filter: OpenAI内容被安全策略拦截"
→ 由 concurrent_pipeline 捕获
→ 匹配安全限制关键词 "filter"
→ 加入 retry_queue（error_type=safety_limit，按单张图片）
→ 主队列完成后在 _process_retry_queue() 阶段重试
```

#### 分支 4.6 - 超时错误（批次级：延迟队列 + 拆分重试）
**触发条件**：翻译器抛出超时异常（如 TimeoutError / asyncio.TimeoutError / httpx.TimeoutException）

```
在 concurrent_pipeline.py 中处理：

批次 [A, B, C] 超时
  → 不做整批重试（避免阻塞主队列槽位）
  → 将 A/B/C 逐张加入 retry_queue（error_type=timeout_error）
  → 主队列完成后，_process_retry_queue() 并行重试（单张图片重试 1 次，受 concurrency/RPM 控制）
```

#### 分支 4.7 - 网络错误（批次级：整批重试 → 延迟队列拆分重试）
**触发条件**：错误信息包含 `connect, reset, 504/524, gateway timeout, cloudflare, <!DOCTYPE html>` 等关键词（timeout 单独归类为“超时错误”）

```
在 concurrent_pipeline.py 中处理：

# 第一次失败：等待3秒后整批重试（不额外等待 RPM 令牌）
日志：网络错误，3秒后整批重试 (1/1)
等待 3 秒
整批重试 [A, B, C]（skip_rate_limiter=True）

IF 成功:
    → 进入阶段 3（成功路径）
ELSE:
    → 将 A/B/C 逐张加入 retry_queue（error_type=network_error）
    → 主队列完成后在 _process_retry_queue() 阶段重试
```

#### 分支 4.8 - 401认证错误（批次级：整批重试 → 延迟队列拆分重试）
**触发条件**：错误信息包含 `401, unauthorized, authentication` 等关键词

```
在 concurrent_pipeline.py 中处理：

# 第一次失败：等待2秒后整批重试（不额外等待 RPM 令牌）
日志：401认证错误，2秒后整批重试 (1/1)
等待 2 秒
整批重试 [A, B, C]（skip_rate_limiter=True）

IF 成功:
    → 进入阶段 3（成功路径）
ELSE:
    → 将 A/B/C 逐张加入 retry_queue（error_type=auth_error）
    → 主队列完成后在 _process_retry_queue() 阶段重试
```

#### 分支 4.9 - 安全限制/数量不匹配/BR缺失/质量失败（批次级：延迟队列拆分重试）
**触发条件**：错误信息匹配安全限制关键词，或翻译器抛出数量不匹配/BR缺失/质量失败异常

```
在 concurrent_pipeline.py 中处理：

批次 [A, B, C] 失败
  → 将 A/B/C 逐张加入 retry_queue（error_type=safety_limit/count_mismatch/br_missing/quality_failed）
  → 主队列完成后，_process_retry_queue() 并行重试（单张图片重试 1 次，受 concurrency/RPM 控制）
```

```
在 manga_translator.py 中处理（HQ翻译器单张失败时）：

单张HQ翻译失败 → 检测是否安全限制错误

IF 是安全限制:
    降级到文本翻译（批量处理，不发送图片）
    openai_hq → openai
    gemini_hq → gemini
ELSE:
    直接抛出异常 → 标记失败
```

#### 分支 4.10 - 其他错误（直接失败）
```
attempt += 1

日志：翻译出错 (attempt/max_retries): {错误信息}

IF attempt >= max_retries (2):
    抛出异常 → 标记失败
ELSE:
    等待 1 秒 → 返回步骤 2.1
```

### 阶段 5：资源释放与完成

**步骤 5.1 - 释放并发槽位**
```
rate_limiter.release()  # 释放 semaphore
→ 其他等待的批次可以获取槽位
```

**步骤 5.2 - 更新统计**
- 成功计数 +1 或 失败计数 +1
- 记录到 `translated_images` 或 `failed_images`

**步骤 5.3 - 并发调整（可选）**
```
IF 最近成功率高:
    rate_limiter.expand_concurrency()  # 扩展并发
    → 触发预热机制（2秒间隔启动新请求）

IF 遇到 429 错误:
    rate_limiter.shrink_concurrency()  # 缩减并发
```

**步骤 5.4 - 标记完成**
- 批次处理完成
- 返回控制权给并发管道

### 总结：重试次数限制

|| 错误类型 | 重试次数（概览） | 备注 |
||---------|----------------|------|
|| 超时错误（timeout_error） | 主队列 1 次（单次超时=400s） + 延迟队列单张 1 次 | 翻译器内部不做超时重试；超时直接入队 |
|| 数量不匹配（count_mismatch） | 翻译器内部重试至 max_retries → 失败后进延迟队列单张 1 次 | 具体次数取决于 `translator.attempts` |
|| 质量检查失败（quality_failed） | 翻译器内部重试至 max_retries → 失败后进延迟队列单张 1 次 | 同上 |
|| BR 标记缺失（br_missing） | 翻译器内部重试至 max_retries → 失败后进延迟队列单张 1 次 | 同上 |
|| **429 限速** | **4 次（含初始）** | 仅翻译器内部处理（独立计数 2/2/4/8s 退避） |
|| 内容过滤（content_filter/safety_limit） | 延迟队列单张 1 次（可能触发 HQ→文本降级） | 延迟队列阶段重试 |
|| 网络错误（network_error） | 主队列整批 2 次（含初始）→ 仍失败后延迟队列单张 1 次 | 整批重试等待 3 秒；延迟队列受并发/RPM 控制；包含 504/Cloudflare/HTML 错误页 |
|| **401认证错误（auth_error）** | 主队列整批 2 次（含初始）→ 仍失败后延迟队列单张 1 次 | 整批重试等待 2 秒；延迟队列受并发/RPM 控制 |
|| 其他错误 | 由 max_retries 控制（一般不进入延迟队列） | 不属于可分类错误时直接失败 |

### 重试次数汇总

|| 错误类型 | 主队列处理 | 延迟队列处理 | 退避时间 |

|| 超时（timeout_error） | 不整批重试，直接入队 | 单张图片重试 1 次 | -（单次请求超时=400s） |

|| 其他错误 | 直接失败或由 max_retries 决定 | - | 1 秒（若有） |
