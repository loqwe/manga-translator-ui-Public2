# 收集者、中转桶与 Worker 的关系说明
这份文档用大白话讲清楚三者的关系、流程，以及单 Worker 和多 Worker 的区别。它对应的是 `concurrent_pipeline.py` 里的实际实现。

## 一句话总览
检测+OCR 把“已可翻译的图片”先放进中转桶，收集者从桶里捞够一批，打包成批次给 Worker 去翻译。Worker 翻译完一批，再取下一批。

## 角色解释（大白话）
1. 中转桶（收集前的中间队列，translation_queue）
这是一个“暂存桶”。检测+OCR 一张图做完，就把这张图的“名字+配置”扔进桶里，表示“这张图可以翻译了”。
它不是最终翻译的队列，只是给收集者准备原料。

2. 收集者（batch_collector）
收集者就是“打包员”。它从中转桶里捞图，一张张收集，凑够一批（比如 5 张）就打包成一个“批次”，再放进批次队列（batch_queue）。

3. Worker（翻译工人，translation_pool_worker）
Worker 就是“翻译工人”。它不直接从中转桶拿图，而是从批次队列拿“一整批”。拿到后就开始翻译，翻完再拿下一批。

## 详细流程（按时间顺序）
1. 检测+OCR 处理图片
- 每张图 OCR 完成后，把“图片名 + 配置”放入中转桶 translation_queue。

2. 收集者开始收集
- 收集者从 translation_queue 里捞图。
- 每捞到一张，就先放到“待打包列表”里。
- 当待打包列表达到批次大小，或者超时/收集结束时，就打包成一个批次。

3. 批次进入 batch_queue
- 打包后的批次被放入 batch_queue。
- batch_queue 是给 Worker 用的，不给检测+OCR 用。

4. Worker 取批次翻译
- Worker 从 batch_queue 拿到批次。
- 设置 busy 标志为 True，然后开始翻译。
- 翻译结束后设置 busy 标志为 False。

## 单 Worker 与多 Worker 的区别
### 单 Worker（translation_concurrency = 1）
- 收集者“放入一批”后，会等待 Worker 翻译完，再收集下一批。
- 这就是串行：一批翻完再收下一批。
- 好处：更稳、更省内存，日志进度更线性。
- 坏处：速度慢一些。

### 多 Worker（translation_concurrency > 1）
- 收集者会持续收集并打包，不必等某个批次翻译完。
- 多个 Worker 并行翻译不同批次。
- 这是流水线模式，吞吐量高。
- 新增规则：如果所有 Worker 都处于 busy 状态，收集者会暂停一下（避免无限堆积）。只要有一个 Worker 空闲，就继续收集。

## 常见疑问
1. 为什么翻译队列剩余是 0，但还在翻译？
因为中转桶只是“收集前的桶”。图片可能已经被收集者捞走，正在等待凑批次或已经在 Worker 翻译。桶空了不代表翻译完成。

2. 为什么多 Worker 时收集者还在收集？
因为多 Worker 是并行流水线模式，收集和翻译本来就是同时进行的。只有当所有 Worker 都忙的时候，收集者才会暂停。

## 总结（核心关系）
- 中转桶：存放“已OCR、可翻译”的图片任务。
- 收集者：从中转桶捞图、打包成批次。
- Worker：只处理“批次”，不碰中转桶。

简单记：中转桶负责“存”，收集者负责“捞和打包”，Worker 负责“翻译”。
