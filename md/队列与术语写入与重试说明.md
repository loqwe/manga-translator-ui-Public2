# 队列与术语写入与重试说明

## 三层队列

### 1) 主队列（检测/OCR → 组批翻译）
- 检测+OCR 顺序执行，结果入 `translation_queue`（翻译）和 `inpaint_queue`（修复）
- 翻译线程按 `batch_size` 组批，等待并发执行
- **卡顿保护**（仅在基线并发=1时生效）：当某个任务卡住主队列时，自动扩展执行槽位
  - 单任务耗时 >200s → 扩槽+1（上限）
  - 扩容冷却 4 分钟；主队列清空后自动收回
  - 修复线程停止后（基线并发>1）不触发
- 代码位置：`concurrent_pipeline.py` 的 `_translation_worker()`

### 2) 并发队列（running_tasks）
- 由 `rate_limiter.concurrency` 控制的并发执行数
- 启动并发=1；修复线程停止后恢复到配置并发（如 1→4）

### 3) 延迟队列（错误重试）
- `retry_queue` 收集：超时、网络错误、401、内容过滤/数量不匹配/BR缺失/质量失败
- 带完整上下文（ctx），重试时只重新翻译，不重做检测/OCR
- 只有主队列和并发队列都清空后才开始处理
- 受当前并发控制，但不阻塞主队列

## 术语写入策略

### 控制变量
术语写入由 `glossary_cache._translation_concurrency` 控制，与 `rate_limiter.concurrency`（翻译执行槽位）**独立**。

### 两种写入模式

**直接写入（并发=1）**
- 启动阶段，修复线程还在运行
- 每次提取术语立即调用 `merge_glossary_to_file()` 落盘
- 代码：`glossary_cache.py` 第392-401行

**延迟写入（并发>1）**
- 修复线程停止后，恢复配置并发时切换
- 术语先缓存到内存（线程安全、自动去重）
- 刷新时机：
  - 每完成偶数个小批量刷新一次（批次2、4、6...）
  - 翻译全部完成时 `flush_all()` 写入剩余
- 代码：`glossary_cache.py` 第402-405行，`on_batch_translation_complete()`

### 卡顿扩槽对术语的影响
- 扩槽时：自动切换术语为延迟刷新模式（并发>1）
- 收回时：如果回到单槽（并发=1），切换回直接写入模式
- 术语写入策略自动跟随实际执行槽位变化

## 错误重试要点

| 错误类型 | 处理方式 |
|---------|----------|
| 429 限速 | 翻译器内部 4 次退避（2/2/4/8秒），不入延迟队列 |
| 超时 | 直接入延迟队列（拆分单张重试），避免卡槽 |
| 网络错误 | 整批重试 1 次（等待3秒）；仍失败入延迟队列 |
| 401认证 | 整批重试 1 次（等待2秒）；仍失败入延迟队列 |
| 安全限制/数量不匹配/BR缺失/质量失败 | 入延迟队列拆分重试，可能触发 HQ→文本降级 |
| 其他错误 | 按 `max_retries` 内部重试；超过后标记失败 |

## 时序流程

```
启动
  │
  ├─ 并发=1，术语直写
  ├─ 修复线程运行中
  │
修复线程停止
  │
  ├─ 并发恢复配置值（如4）
  ├─ 术语切换为延迟写入模式
  │
运行中（可能发生）
  │
  ├─ 卡顿 >200s → 扩槽+1（上限）
  ├─ 冷却 4 分钟
  │
主队列清空
  │
  ├─ 收回临时扩容
  ├─ 处理延迟队列
  │
全部完成
  │
  └─ flush_all() 写入剩余术语
```

## 翻译器超时配置

| 翻译器 | 超时时间 | 与卡顿保护关系 |
|--------|---------|----------------|
| OpenAI HQ | 400s | 会触发（>200s时扩槽） |
| Gemini HQ | 300s | 会触发（>200s时扩槽） |

如果请求卡住超过 200 秒但还没超时，卡顿保护会先触发扩槽。
