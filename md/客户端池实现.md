# 从结构与生命周期完整说明客户端池实现

## 概览
客户端池用于 OpenAI 兼容请求的连接复用与并发隔离。它以“事件循环 + API Key + Base URL + 请求头指纹”为粒度创建池，每个池由多个独立的 httpx 客户端组成。请求在池内轮询分配，超时会主动断开对应连接并在下一次请求时重建。

## 数据结构与缓存键
客户端池对应 `_OpenAIClientPool`，内部维护两组平行列表：`_clients`（AsyncOpenAI 实例）与 `_httpx_clients`（httpx.AsyncClient 实例）。轮询指针 `_next_index` 决定下一次请求落到哪个槽位。

池缓存键由以下四部分组成：
- 事件循环 id（避免跨事件循环共享）
- 规范化后的 base_url（去空格与尾部斜杠）
- api_key
- headers 指纹（稳定序列化后哈希）

这保证了不同用户/不同地址/不同请求头不会共享同一池。

## 生命周期
池的生命周期分为四步。

第一步是创建。第一次请求触发 `_get_openai_client_for_request`，若缓存未命中则初始化一个池。

第二步是扩容。每次请求都会携带期望的 `client_pool_size`（由 `rate_limiter.concurrency` 提供）。如果池长度不足，会逐个创建新客户端并写入池中。

第三步是使用与轮询。请求通过 `_next_index` 取模分配槽位，然后推进指针，形成轮询。只要池大小达到并发数，请求就会落在不同客户端上。

第四步是废弃。当 API Key 或 Base URL 发生变化，旧池会被清理，所有 httpx 连接关闭并输出“废弃”日志。

## 扩容与轮询机制
扩容由 `_ensure_size_locked` 完成。它会按需增加新客户端，并在扩容完成后记录扩容日志。轮询由 `get_round_robin` 完成，保证并发请求不会都挤在同一个连接上。

## 超时策略（总超时与流式读超时）
超时策略统一由 `_build_request_timeout` 构建：
- 总超时为 360 秒（6 分钟），流式与非流式一致。
- 连接超时为 60 秒。
- 流式请求的 read 超时为 60 秒（单 chunk 60 秒无数据即超时）。
- 非流式的 read 超时为 360 秒，与总超时一致。

## 超时断连与重建
当请求触发超时，`chat_completions` 会判定为 timeout 并调用 `pool.mark_timeout`。该函数会把对应槽位标记为不可用，主动关闭旧的 httpx 连接，并输出断连与关闭日志。下一次请求轮询到该槽位时，会自动重建客户端并继续使用。

## 池释放与切换
当用户级 API Key 或 Base URL 改变时，会触发 `close_openai_clients`：
- 从池缓存中移除旧池。
- 关闭所有 httpx 连接。
- 输出“已废弃 X 个客户端”的日志。

## 日志对照与含义
以下日志用于判断池状态与异常处理是否生效：

```
[客户端池] 创建新客户端 #N
[客户端池] 已扩展: 1 -> N 个客户端
[超时] 已断开旧连接，下次请求将重建客户端
[客户端池] 超时后已关闭旧连接
[客户端池] 已废弃 X 个客户端
```

## 如何在日志里判断并发扩容是否生效
判断是否扩容生效，优先看这些关键日志是否按顺序出现，并且数字符合预期的并发数。

第一层是“池扩容是否发生”。当并发数从 1 提升到 N 时，应看到“已扩展”日志，然后出现新增槽位的“创建新客户端”日志。这说明池长度已经扩到目标值。

第二层是“并发控制是否放开”。并发流水线在修复线程结束后，会有“修复线程已停止”和“并发数扩展”的日志，代表并发限制从 1 恢复到配置值。

第三层是“运行中并发是否提升”。翻译线程会打印“运行中: x/y”，当 y 大于 1 且同时存在多个运行任务，说明并发实际生效。

## 如何排查“池大小不足导致串行”
若观察到请求始终串行，可按这个顺序排查。

第一步确认池是否真的扩到了并发数。如果只看到“创建新客户端 #1”而没有“已扩展: 1 -> N”，说明池仍是 1。常见原因是 `client_pool_size` 计算结果仍为 1。

第二步确认并发限制是否放开。并发流水线在修复线程未结束前会强制串行，如果没有“修复线程已停止”和“并发数扩展”的日志，串行是预期行为，不是池的问题。

第三步确认是否缺少 `rate_limiter` 绑定。池大小来自 `ctx.rate_limiter.concurrency`，如果当前流程没有绑定 rate_limiter，池大小会回退为 1，即使外部设置了并发也不会扩容。

如果扩容日志齐全、池大小已扩大但仍表现串行，通常不是池的问题，而是上层并发控制、队列等待或 RPM 限制在起作用，需要结合翻译线程“运行中任务数”和速率限制日志一起判断。
