# OpenAI 网关模块说明文档（详细版）

## 一、调用链路怎么走（白话版）
上层翻译器（普通翻译或高质量翻译）在需要请求 OpenAI 时，不再自己创建客户端，而是调用网关里的 `chat_completions`。网关会先检查是否已经有“当前事件循环”的共享客户端，如果有就复用，没有就创建并缓存。然后网关根据你设置的参数组装请求，发出调用，最后把结果统一整理成一个结构化结果对象返回给上层。

换句话说，翻译器只管“要翻译什么”，网关只管“怎么打电话、怎么复用线路、怎么统计”。

## 二、连接复用是怎么做的
网关内部有一个缓存表，以“事件循环 + API 地址 + API Key + 请求头指纹”为键，存放共享的 AsyncOpenAI 客户端和对应的 httpx 连接池。这样做的结果是：

同一个事件循环内，多次请求不会反复新建客户端和 TCP/TLS 连接，而是复用已有连接池。HTTP/2 也在这里开启，让同一条连接上可以并发多路请求，减少握手和队头阻塞的概率。

这就是“连接复用”的核心：不是每次翻译都新建网络通道，而是复用一条高质量通道。

## 三、限速器（节流）是怎么做的
网关提供了一个轻量的 RPM 限速器。它按 “base_url + model” 作为键记录上次请求时间，然后计算最小间隔：

- 如果设置 `max_requests_per_minute`，它会把 60 秒除以 RPM 得到最小间隔时间。
- 每次发请求前检查“距离上次请求”是否小于这个间隔，如果小于就 `sleep` 一会儿再发。
- 发完请求后会更新这次请求时间。

这类限速属于“软节流”，重点是避免“太快导致 429”，而不是强制排队到极致。节流逻辑在网关内完成，上层调用不需要关心。

## 四、结果对象里有什么
`chat_completions` 返回的是统一的 `OpenAIChatResult`，里面包含：

- `response`：原始 OpenAI 响应对象（非流式）
- `text`：提取出的文本结果（流式或非流式统一）
- `usage`：token 使用情况（如果有）
- `request_id`：请求 ID
- `finish_reason`：结束原因（如 stop、length、content_filter）
- `stream`：是否是流式
- `send_ts / recv_ts / duration_ms / first_byte_ms`：时间指标
- `status`：ok / timeout / rate_limit / content_filter / error

上层翻译器只需要取 `text` 和必要的元信息，日志和指标可以统一从这里打点。

## 五、和“改之前”的主要区别
以前的做法是每个翻译器自己创建客户端、自己处理连接细节。结果是：

- 每次请求都可能重新建连接，网络握手多、延迟高。
- 并发时会出现多个客户端争抢资源，连接池无法复用。
- 流式与非流式的处理逻辑分散在各处，难以统一维护。
- 记录指标与统计耗时更碎片化。

现在的做法是：

- 客户端创建集中在网关里，同一事件循环复用同一连接池。
- HTTP/2 在网关里统一开启，连接更少、吞吐更高。
- 限速器在网关里统一执行，避免一处快一处慢。
- 结果对象统一，方便上层处理和日志统计。

一句话总结：以前是“每个翻译器自己打电话”，现在是“通过网关统一拨号、统一线路、统一话单”。

## 六、典型调用顺序（简化版）
1. 翻译器调用 `chat_completions(...)`
2. 网关获取或创建共享客户端
3. 如果设置了 RPM，先做节流等待
4. 发送请求（流式或非流式）
5. 收集文本与指标，包装成统一结果对象
6. 返回给翻译器

以上流程保证了连接复用、节流控制与结果结构的一致性。
